## 项目：LangGraph 营销内容生成系统

---

### Q1 为什么用多Agent而不是单Agent？

单Agent技术上能做大部分事：调多个模型、实现ReAct/CoT、写循环重试、手写断点恢复都可以。真正不可替代的差异只有两条：

**① 并行执行**：单Agent是单线程，步骤必须顺序等待。多Agent可以让多个节点同时运行，物理上单Agent做不到。

**② 评估独立性**：Evaluate Agent打分必须只看最终输出，不能看到生成过程。单Agent的context里积累了研究结果、草稿、重试记录，评估时判断会被污染。多Agent的Evaluate节点只接收最终产出，隔离是架构层面强制保证的。

**③ 上下文积累**：单Agent顺序执行时context不断堆积历史噪声，到后期步骤时模型推理质量下降，极端情况撞上context window上限。多Agent每个节点只接收它需要的字段，context始终干净。

这个项目规模下单Agent够用，选多Agent是因为这些场景在生产环境里是真实需求，同时验证自己能把LangGraph这套设计落地。

### Q2 Supervisor模式和Pipeline模式有什么区别？为什么选Supervisor？

**Pipeline**：路由顺序固定，条件跳转由代码写死。
```python
def route_after_qc(state):
    if qc_score < 0.7:
        return "script_agent"   # 代码决定
    return "image_agent"
```

**Supervisor**：有一个LLM作为调度节点，每步完成后由LLM读当前状态、自己推理决定下一步调哪个Agent，路由逻辑在LLM里，不是硬编码。
```python
def supervisor(state):
    response = llm.invoke(f"当前状态：{state}，下一步调用哪个Agent？")
    return {"next": response.content}  # LLM说去哪就去哪
```

**我的项目实际是Pipeline加条件路由**，不是真正的Supervisor模式。这是有意的选择：内容生成流程清晰、质量门控标准明确，条件路由更可控更稳定。真正的Supervisor适合任务边界模糊、需要LLM灵活调度的开放式场景。

### Q3 并行执行适合什么场景？Supervisor+并行什么时候用？

**判断要不要并行，看两个条件**：任务之间互相独立 + 主要时间花在等待外部响应。

**我的项目为什么不用并行**：给一个任务输出多种结果（三平台文案），一次LLM调用批量生成即可，不需要并行。

**什么时候用并行**：同时来了十个独立任务，每个任务派一个Agent并行处理，不互相等待。一个任务输出多种结果 → 不需要并行；十个独立任务 → 适合并行。

**Supervisor+并行适合的场景**：任务边界在运行前不确定、子任务互相独立。例如用户问"分析特斯拉的市场、技术、财务风险"，Supervisor运行时决定拆成三个子任务，同时派三个Agent并行分析，最后汇总。核心是：拆成几个、拆什么方向，运行时才知道。

### Q4 QC不达标会回退Script Agent，这个回退逻辑怎么实现的？最多2次重试是硬编码还是可配置的？

**第一条：当前实现思路**

检查两个规则：
1. 必须生成3套文案
2. 每套文案body长度超过50字

路由逻辑：两条都满足 → 进入Image Agent；不满足且重试次数未到2次 → 回退Script Agent重新生成；不满足但已重试2次 → 强制放行进入Image Agent，防止无限循环。

重试计数器retry_count存在GraphState里，每次不通过自增1，路由函数读这个值做判断。

**最多2次是硬编码**：上限直接写在路由函数的条件判断里，不可配置。

---

**第二条：如果面试官觉得规则太浅，可以从这几个方向扩展**

1. **关键词覆盖检查**：验证Research Agent搜到的趋势关键词是否出现在文案里
2. **平台字数限制**：微博正文不超过140字、朋友圈不超过150字，加平台专属规则
3. **必要元素检查**：小红书必须有emoji、所有平台必须有hashtag
4. **LLM-as-Judge替代规则层**：用模型打分替代硬规则，能判断语义质量，但成本和延迟会上升

### Q5 为什么用LangGraph而不是LangChain普通链？LangGraph解决了什么问题？

**第一条：LangChain和LangGraph没有硬性区别**

LangGraph是建在LangChain基础上的，LangChain加自定义代码可以实现LangGraph的所有功能。选LangGraph不是因为非它不可，而是它把复杂流程封装好了，不用自己造轮子。

---

**第二条：LangGraph相比LangChain提供的框架级能力**

1. **有向图结构**：用节点+边定义流程，支持循环、分支、条件路由，流程逻辑写在图定义里，清晰可读
2. **集中式GraphState**：所有节点共享同一个状态对象，字段有类型约束，数据流向明确
3. **条件路由**：add_conditional_edges根据State里的值动态决定下一步去哪个节点
4. **内置Checkpoint断点恢复**：每个节点执行完自动存快照，中途失败从上一个成功节点继续
5. **并行执行**：Send机制同时触发多个节点并行运行
6. **Human-in-the-loop**：原生支持interrupt，任意节点暂停等待人工审核后继续
7. **流程可视化**：图结构可以直接生成可视化图，方便调试

---

**第三条：为什么这个项目选LangGraph**

项目需要QC不达标回退Script的循环结构和断点恢复能力。这两个功能用LangChain自己实现需要写大量的状态管理和流程控制代码，繁琐且容易出错。既然有现成的框架可以直接支持，就选择使用现有的框架。

### Q6 MCP是什么协议？为什么用MCP工具而不是直接写Python函数？

**第一条：三种调用方式的区别**

**直接Python函数调用**：工具逻辑和agent逻辑写在同一份代码里，完全耦合，工具变了agent代码就得改。

**Function Calling**：调用由大模型决定，不再是硬编码顺序调用，但工具的定义和实现仍然写在agent代码里，耦合没有解决。

**MCP（Model Context Protocol）**：工具逻辑完全独立，写在单独的server脚本里，agent代码里没有任何工具实现。大模型通过MCP协议发现和调用工具，agent层和工具层彻底解耦。

---

**第二条：解耦带来的具体好处**

1. **工具可以复用**：同一个MCP server里的工具可以被多个agent调用，不需要在每个agent里重复定义
2. **独立维护**：工具逻辑更新不影响agent代码，agent逻辑更新不影响工具
3. **标准化协议**：工具的描述、参数、返回格式统一，便于大模型理解和调用

---

**第三条：MCP工具和Skill的区别**

**MCP工具**是原子级的单一函数，做一件具体的事：web_search就是搜索一次，generate_image就是生成一张图，save_result就是保存文件。

**Skill**是agent封装的高层能力。比如Research Agent的skill是"找市场趋势热词"，它会多次调用web_search工具、对比结果、提炼关键词，这一整套能力是它的skill。Script Agent的skill是"写营销文案"，不需要调用任何MCP工具，纯靠LLM推理完成。

在Supervisor模式里，每个Agent封装一种skill，Supervisor知道有哪些skill可用，根据当前状态决定调哪个。

层级关系：**MCP工具（原子函数）→ Skill/Agent（高层能力）→ Supervisor（调度skill）**

### Q7 LLM-as-Judge可靠吗？有没有可能模型自己评自己打高分？

**第一条：LLM-as-Judge的已知问题**

1. **自评偏差**：用同一个模型生成内容再评估，模型倾向于给自己的输出打高分
2. **冗长偏差**：模型倾向于给更长的输出打高分，即使质量没有更好
3. **不一致性**：同样的内容多次评估可能得到不同的分数
4. **对齐偏差**：模型的审美偏好不一定和真实用户偏好一致

---

**第二条：我的项目怎么处理自评偏差**

文案生成用DeepSeek-V3，图片分析用gpt-4o-mini，评估用Qwen-VL-Max，生成和评估使用完全不同的模型，从根本上避免自评偏差。

---

**第三条：如果要进一步提升可靠性**

1. **结构化评分标准**：给评估模型明确的rubric，比如关键词覆盖率、平台风格匹配度，避免模型凭感觉打分
2. **多次评估取均值**：同一内容评估多次，降低单次评估的随机性
3. **Human-in-the-loop**：关键决策引入人工审核做最终校验

### Q8 图文一致性怎么评估的？Qwen-VL-Max的prompt怎么设计的？

**第一条：三个评估维度**

1. **文案质量**：语言表达是否流畅、是否符合平台风格，1-10分
2. **图片合规性**：画面是否包含违规或敏感元素，1-10分
3. **图文一致性**：图片风格是否与文案主题匹配，1-10分

---

**第二条：prompt设计思路**

把三套文案拼成可读文字，和图片（base64编码）一起放进同一条消息里，让Qwen-VL-Max同时看到文字和图片，一次调用完成三个维度的评估。每个维度要求给出分数和简短理由，避免模型只给分不解释。temperature设为0.1，降低评估随机性，让结果更稳定一致。

---

**第三条：为什么只取2张图**

图片转成base64之后体积很大，一张图可能消耗几千到上万token。Qwen-VL-Max的context window是128K token（131,072 token），最大输出是32K token（32,768 token）。虽然上限较大，但为了控制成本和调用速度，代码里限制最多传2张图。

---

**第四条：为什么选Qwen-VL-Max**

需要同时理解文字和图片，必须用多模态模型。Qwen-VL-Max支持图片输入，并且和生成模型DeepSeek-V3完全不同，保证评估独立性，避免自评偏差。
